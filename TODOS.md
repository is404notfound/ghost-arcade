## 캐릭터 선택 + 캐릭터별 스탯 차등 (론칭 후, 데이터 기반)
- **What:** 게임 시작 시 달리는 캐릭터 선택. 캐릭터별 점프 횟수 / 체력 / 속도 차등.
- **Why:** 리텐션/깊이 기능. 핵심 루프엔 불필요(단일 캐릭터로 Go/No-Go 통과, 8연속 플레이).
- **When:** 론칭 후 실데이터로 밸런스 → 시즌 콘텐츠("새 캐릭터 출시")로. 지금 하면 백엔드가 밀리고 밸런스 데이터도 없음.
- **성격 분리:**
  - 외형만 선택 = 연출(버전 무관, 에셋 필요) → 에셋 단계 가능.
  - 스탯 차등 = 메커니즘(sim, 버전 업). 고스트 재생 위해 로그에 캐릭터 정보 필요.
- **리더보드 공정성 → (나)로 해소:** 스탯 차이는 유지하되 **토큰 보상 경쟁을 티어 분리**(토큰 경제 섹션). 캐주얼은 단일 보드 유지. 속도 차등이 점수 직결이라 티어 분리가 특히 중요 — 티어는 로드아웃 전력 기준.
- **고스트 렌더 결정(2026-06-18):** 고스트는 캐릭터 무관 **한 디자인으로 통일 렌더**(에셋 절감 + 시각 정리). 단 "똑같이 보이기 ≠ 똑같이 움직이기" — 로그엔 characterId를 담아 **궤적은 그 캐릭터 스탯대로 재생**해야 함(안 그러면 desync). 렌더만 무시.
- **Forward-design (지금 거의 공짜, 백엔드 로그 스키마 설계 시):** 입력 로그 스키마에 `characterId` 슬롯 예약(기본값 'base'), `parseLog`은 필드 없으면 'base'로 관대하게. → 나중에 캐릭터 추가가 *가법적*(기존 스탯 불변 시 버전 업 없이 추가). 안 해두면 추가 시점에 그때의 실유저 고스트 전부 무효화.
- **Depends on:** 론칭 + PostHog 실데이터. Forward-design만 백엔드 스키마 설계 때.

- **성격:** 메타데이터라 sim 무관(버전 업 없음). 단 *남이 봐야 의미* → 크로스유저(백엔드)가 있어야 빛남. 고스트 통일 디자인과 안 부딪힘(닉네임은 라벨, 스프라이트 아님).
- **When:** 백엔드/리더보드와 함께. **Forward-design:** 레코드 스키마에 `nickname` 필드.

## 부스터 아이템 (버킷 C — 론칭 후, 데이터 기반)
- **What:** 게임 시작 전 **1회 선택**(중간 소비 아님). 종류: 초반 자동달리기, 체력바 늘리기, 먹는 체력 증가, 3회 장애물 무적, **포션 자석(근처 포션 자동 흡입)**.
  - 자석 근거(2026-06-24): 테스터 공통 피드백 "힐팩 위치 애매해 못 먹음" → 기본 게임은 포션을 1단 점프 도달 범위로 낮춰 해결(SIM_VERSION 업). 자석은 그걸 *유료 편의*로 전환 — 높은/먼 포션도 끌어와 먹게 하는 부스터 매력. 도달성 문제를 부스터 가치로 재활용.
- **성격:** 메커니즘(sim, 버전 업). 고스트 재생 위해 *그 판의 부스터*를 로그에 기록 필요(`modifiers`).
- **결정 갱신(office-hours 2회차):** 무토큰 **캐주얼은 단일 보드 유지**("그게 경쟁"). 단 **토큰 보상 play는 티어 분리(나)** — pay-to-win 방지(토큰 경제 섹션). 부스터는 **토큰 sink** 역할.
- **경제:** 1회 선택이라 중간 인벤토리/실시간 획득 불필요(가벼움). 획득 방식(무료/해금/유료)은 추후 결정.
- **When:** 론칭 후. 밸런스·가용성은 실데이터로(검증 안 됨, 핵심 루프엔 불필요).

## 주간 누적 거리 랭킹 (게임오버 패널, 버킷 B — 백엔드/리더보드 단계)
- **What:** 게임 종료 후 결과 패널에서 "일주일간 달린 총 거리" 기준 랭킹을 보여준다.
  (현재는 단판 거리 + 고스트 추월 비교만 표시.)
- **Why:** 단판 성적이 아닌 *누적 플레이*를 보상 → 리텐션 동기(매일 와서 거리 쌓기).
  데일리 시드라 매일 다른 코스를 달려도 주간 누적은 공정한 꾸준함 지표.
- **데이터:** 백엔드가 `ghost_runs`의 distance를 **유저별·주별로 합산**해야 함(현재 스키마는
  per-(시드,런) 저장이라 주간 집계 쿼리 또는 별도 집계 테이블 필요). 닉네임/playerId(4-1)와
  연계돼야 "누구의 주간 누적"인지 식별 가능 → Forward-design meta.nickname/playerId 슬롯 활용.
- **경계:** 집계/표시라 sim 무관(버전 업 없음). 크로스유저 데이터 + 주 경계(UTC 월요일 등) 정의 필요.
- **When:** 백엔드/리더보드 단계. 닉네임(4-1) 이후. 그 전까지는 단판 거리 비교 유지.

## 사망 지점 분포 그래프 (버킷 B — 백엔드/리더보드 단계)
- **What:** 죽었을 때, "사람들이 어느 거리에서 죽었는지" 분포(히스토그램)를 보여주고 내 위치를 표시. "당신은 상위 N%" 류 피드백.
- **성격:** 분석/표시 기능 — sim 무관(버전 업 없음). 단 *전체 유저의 사망 거리 집계*가 필요 → 백엔드/크로스유저 데이터 의존. 셀프 데이터만으론 무의미.
- **데이터 출처:** 거의 공짜 — 고스트/리더보드가 이미 각 판의 `distance`를 저장하므로, 백엔드가 그걸 집계만 하면 됨(추가 수집 불필요).
- **분포 모양 주의:** 깔끔한 정규분포(종형)는 **아닐 가능성이 높음.**
  - 보통 **오른쪽 꼬리(우편향)** — 초반에 죽는 사람 多(학습/이탈/짧은 시도) + 멀리 가는 소수의 긴 꼬리. 로그정규/지수 감쇠에 가까움.
  - **특정 거리에 사망 스파이크** — 데일리 시드라 모두 같은 코스 → 어려운 패턴('벽')에서 죽음이 몰림. 종형이 아니라 봉우리들이 보임.
- **이중 가치:** (1) 플레이어 동기부여("상위 27%"), (2) **개발 진단** — 사망 스파이크가 "어디가 너무 어렵나"를 알려줌(밸런스 튜닝 근거. 단 sim 튜닝은 론칭 후 = 시즌 묶음).
- **When:** 백엔드/리더보드 단계(집계 데이터 필요). 버전 무관.

## Forward-design — 백엔드 로그/레코드 스키마 슬롯 예약 (백엔드 설계 시, 거의 공짜)
- **What:** 입력 로그/리더보드 레코드 스키마에 `characterId`(기본 'base'), `nickname`, `modifiers`(부스터) 필드를 미리 비워둠.
- **Why:** 그러면 나중에 캐릭터/닉네임/부스터를 **버전 업·고스트 무효화 없이 가법적으로** 추가 가능. 안 해두면 추가 시점의 실유저 고스트가 전부 날아감. (version 태그를 미리 박은 것과 같은 싼 보험)
- `parseLog`은 없는 필드를 기본값으로 관대하게 처리(이전/타버전 로그 호환).
- **When:** 백엔드 스키마 설계 시 (다음 대형 작업). 이것만 백엔드 단계, 나머지 기능은 그 후.
