# 리텐션 개선 계획 — 바이브코딩 챌린지 대응

작성일: 2026-07-22
상태: 의사결정 대기 (일부 항목 완료)
근거: 실서비스 PostHog 데이터(라이브 D1, 2026-07-21~22) + 코드 확인

## 1. 메인 목표

**토스 바이브코딩 챌린지 1차 심사가 8/1~26 실사용 지표 기준**이다. 8월 전에 리텐션을
확보해서, 이 한 달간 지표를 최대치로 뽑는 것이 목표.

**의사결정 필요:** 어떤 지표를 개선 대상으로 삼고, 어떻게 개선할지.

### 현재까지 확인된 근거 (2026-07-22 기준, 라이브 D1~D2)

- **`game_start` 이벤트 ~83건, 고유 유저 ~36~40명** — 라이브 약 30시간(7/21T00:06~
  7/22T05:50) 동안. 지난 설계 문서의 킬러쿼리 표본 문턱(100판)에 하루 만에 80% 근접 —
  자연 유입은 예상보다 건강함.
- **스킬 커브 확인됨(하드월 아님):** `lifetime_run_index`(생애 몇 번째 판)별 평균
  `distance`가 첫 1~3판 ~220~620m → 10~13번째 판 ~1,472m로 **단조 상승**. 반복하면
  느는 정상적인 곡선이지, 특정 지점에서 못 넘는 벽이 아님.
- **하지만 이 스킬 커브가 느리다.** 10판 넘게 반복해야 1,400m대에 도달하는데, 봇 고스트
  하위 티어가 `[250, 700, 1400, 2400, 3500, 4800, 6200, 7800]`(`src/botRecorder.ts:272`)
  이고, 고스트 필드는 "실유저+봇 기록 합산 후 거리 내림차순 top-8"이라(`GameScene.ts`
  `mergeGhostRecords`), 실유저 기록이 낮으면 top-8 대부분이 봇 상위 티어(1400m+)로
  채워진다. **결과: 신규 유저가 "고스트를 한 번이라도 이겨보는" 경험을 하려면 최소
  10판 넘게 반복해야 할 가능성이 높음.** 이 게임의 핵심 재미 가설("고스트 제침")에
  도달하는 시간이 너무 길다는 뜻.
- **사망 원인 분포(`game_over` count by `death_cause`/`death_obstacle_height`):**
  - `hp_drain`(체력 자연 감소사) — 전체 사망의 **~37%**, 단일 최대 사인. 장애물과 무관.
  - 충돌사 중 높이 139~148 구간(= `OBS_PATTERNS`의 **TALL, h=148**, "더블 점프 권장"
    패턴)이 최다. 더블 점프 미숙련 신규 유저가 집중적으로 여기서 죽는 것으로 추정.
- **`overtakes` 분포는 아직 미확인** — 위 가설(고스트 필드 도달 불가)의 직접 검증 필요.

## 2. 업데이트 필요사항

### 2.1 고스트 경쟁감 강화
고스트가 "이미 달렸던 실유저"라는 느낌을 지금보다 더 살려서, 경쟁하고 있다는 감각을
강화해야 함.

**우선 보강 방향 — 제침(고스트 사망) 순간 강조:**
고스트가 죽거나 뒤처지는 순간을 지금보다 명확히 연출하면 "내가 실제 경쟁자를 제쳤다"는
쾌감이 살아난다. 후보:
- 고스트가 죽을 때 이름표(닉네임) + 최종 거리 팝업/플로팅 텍스트 ("시안여우-42 제침!").
- 제친 순간 짧은 이펙트(파티클/플래시/사운드)와 가벼운 카메라 반응.
- HUD에 실시간 순위 변동 표시("N위 → N-1위") 강조.
- 제친 상대가 "실유저였음"을 드러내는 표기(봇/실유저 구분 배지 등).

→ **2.4 사다리 수정으로 제침이 자주 발생하게 됐으므로(median 유저 판당 ~3제침),
이 연출은 발동 빈도가 높아져 리텐션 레버로서 효과가 크다.** 순수 렌더 연출이라
`sim/`(SIM_VERSION) 무관·저위험.

**설계 확정 (2026-07-24, office-hours) → `docs/design-overtake-juice.md`:**
- **계단식(Tier 1/2/3):** 매 제침마다 전부가 아니라 무게에 비례. Tier1(일상 단일 제침)=
  닉네임+거리 팝업·고스트 힐끗·sfx 피치·HUD 순위 틱 / Tier2(연쇄2·top-3 진입)=플래시·
  파티클·punchZoom / Tier3(연쇄3+·1위 탈환)=강화+격추 로그. 이유: 판당 제침 다발 →
  무차별 풀연출은 피로·모바일 프레임드랍·닷지 방해를 부름.
- **슬로모 배제:** 결정론 sim/고스트 lockstep을 흔들어서 안 씀. "큰 순간"은 플래시 +
  기존 `punchZoom()` + 파티클로 대체.
- **실유저/봇 배지 없음:** 봇도 실유저처럼 균일 대우 → "전부 사람" 환상 유지.
- **구현 순서:** Tier1 먼저(저위험·빈번 케이스 커버) → streak+Tier2 → Tier3.

### 2.2 유저 아이디 PostHog 수집 — ✅ 완료
`posthog.identify(getUserId())`를 부팅 시 호출해 PostHog distinct_id를 우리 영속
익명 UUID로 고정 완료(`517f1be`, 2026-07-22 새벽, `src/analytics/index.ts`
`identifyUser()` / `src/main.ts:60`에서 `initAnalytics()` 직후 호출). PostHog·
Supabase·토스가 같은 키를 공유해 크로스소스 조인 가능.

### 2.3 튜토리얼 강화
데이터로 확인된 근거(위 §1 참조):
- 너무 일찍 죽는 유저가 많음 (`lifetime_run_index` 1~3판 평균 거리 ~220~620m).
- 주된 사인은 `hp_drain`(37%) — 포션 획득/피버 발동을 통한 체력 관리 학습 부족 추정.
- 충돌사는 **TALL 패턴(h=148, 더블 점프 요구)** 에 집중.

→ 튜토리얼이 "체력 관리(포션 우선순위)"와 "더블 점프 타이밍"을 신규 유저에게 더
명시적으로 가르쳐야 할 근거가 됨.

**설계 확정 (2026-07-24, office-hours) → `docs/design-tutorial-onboarding.md`:**
- **핵심 통찰: "난이도 죽음"이 아니라 "지식 죽음".** 어려워서가 아니라 포션 회복·더블점프·
  피버 탭회복을 *몰라서* 죽음 → 가르치기(렌더 토스트)만으로 상당 부분 해결 가능.
- **Part A (렌더 전용, 저위험, 먼저):** 첫 만남 1회성 토스트 2종 — 포션("회복!"), 더블점프
  ("↑↑ 두 번 탭", 첫 TALL). localStorage 플래그, 비차단. sim 무관·리셋 불필요. §2.1과 함께 출시.
- **Part A-2 (피버 상시 촉구, 렌더 전용):** 피버 내내 영문 "tap!"을 산발적으로 띄웠다
  사라뜨리길 반복 → "계속 두드려" 촉구. 매 피버마다(일회성 아님). 지식+CTA+주스 겸용.
- **Part B (sim 난이도 커브, 조건부):** 전원 공유 코스 첫 ~300m 완만화. 리더보드 리셋 허용됨
  (한 번 갈아엎을 예정)이라 가능해졌으나 `SIM_VERSION`↑ → **모든 sim 변경을 단일 1.15.0으로
  묶어 8월 1일 전 1회 리셋**(챌린지 창 중 리셋 금지). Part A 측정 후 필요할 때만.
- **피버 미활용 원인 확정:** 발동은 잘 됨(콤보10 도달 문제 아님). 진짜 격차는 "피버 중
  계속 탭해야 회복"을 모르는 것 → Part A-2 상시 촉구로 해결. **sim 튜닝(콤보 하향) 불필요.**
- **지표:** `death_cause` hp_drain%·TALL%, `fever_count`, 신규 평균거리·D1.

### 2.4 고스트 필드 "실력 사다리" 선정 — ✅ 구현 (플레이테스트/배포 대기, 2026-07-23)

**진단 확정(라이브 데이터):** 전체 156판 중 **74%가 0제침**. 원인은 봇 티어가 아니라
**필드 선정 방식**이었다:
- `remoteStore.loadTopRunsRemote`가 `.order(distance desc).limit(8)` → 원격에서 *가장
  멀리 간 8판만* 가져오고, `mergeGhostRecords`가 다시 거리순 top-8로 자른다.
- 실유저 기록이 쌓이면서 하위 봇(250·700m)은 top-8 밖으로 밀려났고, 필드 바닥이
  1,400~1,600m로 굳었다. median 거리 576m 유저는 구조적으로 아무도 못 넘는다.
- 즉 "봇 타깃 재조정"은 무의미(낮춰도 top-8에서 또 빠짐). 넘을 수 있는 저거리 **실기록은
  DB에 넘치는데**(절반이 576m 미만) top-8이 통째로 버리고 있었던 게 진짜 병목.

**처방(구현됨):** 필드를 "거리순 top-8"이 아니라 **풀 전체의 백분위 사다리**로 뽑는다.
- `ghostStore.selectLadder(sortedDesc, n)` — 풀에 고르게 퍼진 8칸(idx0=최장 엘리트 →
  하단 ~하위 8% 발판). 순수 함수, 단위 테스트 6종.
- `loadTopRunsRemote` limit 8 → **150(LADDER_POOL_N)**: 사다리 하단 발판용 저거리 표본 확보.
- `GameScene`: `mergePool`(상한 없는 병합) 분리, `applyGhostLadder`(필드=사다리, 칩=진짜
  top3) 추가. **`mergeGhostRecords`(랭킹 top-8)는 불변** → 게임오버 랭킹 패널 영향 없음.

**실데이터 검증(오늘 시드 70판):** 구 필드 `[5924,5093,4384,2866,2374,2259,1601,1576]`
(바닥 1576m, median 0제침) → 신 사다리 `[5924,1349,889,807,588,334,229,120]`
(median 588m 유저가 3칸 제침). 봇 없이 실유저 기록만으로 초반 제침 열림.

**속성:** 렌더/데이터 레이어 — `sim/`(SIM_VERSION) 무관, GRAC 밸런스 신고 대상 아님.
인구 늘수록 사다리 자동 재조정.

**후속(이번 범위 아님):** 사다리를 *플레이어 개인 실력*(best distance)에 맞추는 스킬밴딩,
그리고 일 볼륨 폭증 시 서버측 백분위 샘플링(RPC).
