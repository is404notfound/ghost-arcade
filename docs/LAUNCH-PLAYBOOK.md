## 0. D-minus 구간 — 심사 대기 중인 지금만 할 수 있는 일

검토에서 나온 대비책 대부분이 **지금** 해야 하는 작업이다. 승인이 나면 늦다.

- [ ] **계측 E2E 리허설**: QR 테스트 빌드로 실기기 토스앱에서 시나리오 1회(진입→첫판→사망→재시도→종료) 수행, PostHog Live Events에 이벤트가 **속성까지 정확히**(특히 `is_retry`, 부트 단계) 도착하는지 눈으로 대조
- [ ] **PostHog 키 off 리스크 해소**: 검수 시 PostHog 허용 여부 확정. 애매하면 — `game_start`/`game_over`/user_id를 Supabase에 미러링하는 이중화를 지금 구현 (키 off 출시 시 §1을 Supabase 프록시 지표로 대체하는 축소판 기준표를 부록 A로 사전 커밋)
- [ ] **게임오버 이벤트 유실 방지**: `game_over`는 즉시 플러시/`sendBeacon` 경로로 — 웹뷰를 게임오버 직후 닫는 게 미니앱 표준 종료 패턴이라, 안 하면 완료율·사망 분포가 체계적으로 과소측정
- [ ] **비정상 종료 탐지**: localStorage 하트비트(시작 시 기록, 정상 종료 시 해제, 다음 부트에 잔존하면 `abnormal_exit` 송신) — 저사양 웹뷰 OOM은 JS 예외가 아니라 Sentry에 안 잡힌다
- [ ] **원격 킬스위치**: 재검수 없이 조정 가능한 수단 확보 — Supabase 원격 config(정전 트랩 on/off·난이도 계수, 기능 플래그 2~3개). 앱인토스 긴급 검수 fast-track 존재 여부도 채널톡으로 확인해 여기 기록: ______
- [ ] **리더보드 오염 즉응**: 이상치 임계(물리 최대 거리 + 봇 최고기록 ×1.5) 사전 커밋 + 격리 SQL 1개 작성 — 새벽에 1등이 999,999m면 아침 30분 안에 숨길 수 있어야 함
- [ ] **PostHog 코호트 리허설**: 테스트 user_id 2개로 static cohort 생성 → 리텐션 매트릭스에 "제외" 필터 적용 → 대시보드 저장까지 (person profiles 모드 확인 포함 — 익명 이벤트 모드면 §5 전체가 무효)
- [ ] **사내 사전 합의**: §1 기준표 + W4 3-way 표를 사업부에 공유하고 "이 기준으로 판정한다" 동의 확보 — 판정 후 협상이 아니라 판정 전 계약 (토큰 경제 thesis와 회수 판정이 충돌할 수 있는 지점)
- [ ] **GRAC 내용수정신고 범위 확인**: 밸런스 패치/보상형 광고 추가가 신고·재분류 대상인지 — W2 직전에 발견하면 타임라인 전체가 밀린다
- [ ] 검수 반려 → 재제출 시: D-0 체크리스트를 처음부터 재실행 (반려 기간 중 환경 변화 가능)

## 1. 판정 기준표 (사전 등록)

기준선 근거: 하이퍼캐주얼 D1 평균 ~26–28%, 상위 38–40%, D7 ~8%
([AppAgent](https://appagent.com/blog/mobile-game-retention-benchmarks/),
[GameAnalytics 2025](https://www.gameanalytics.com/reports/2025-mobile-gaming-benchmarks)).
미니앱(설치 마찰 없음 = 유입 질 낮음)만큼 **일관되게 ~25% 하향 보정**.

**판정 게이트**: ① 모든 % 지표는 오가닉 코호트(§5) 기준, n<100 판정 유보(n<30은 추세 언급도 금지)
② **기술 지표(크래시·로드·고스트%) 중 "성공" 미달이 있으면 게임성 지표는 "조건부"로만 기록** —
기술 정상화 이후 유입 코호트로 재판정. 느려서 떠난 걸 "코어 루프 반증"으로 오판하는 게 최악의 학습 오염.

| 지표 | 정의 (분모 명시) | 성공 | 보통 | 실패 | 판정일 |
|---|---|---|---|---|---|
| 진입→첫 판 전환율 | `game_start` 있는 유저 / 부트 진입 유저 ※생존편향 각주1 | ≥80% | 60–80% | <60% | D7 |
| 첫 판 완료율 | 첫 `game_over` / 첫 `game_start` (유저 단위) | ≥90% | 75–90% | <75% | D7 |
| **자발 재시도율 (주지표)** | **첫 세션에서 2판 이상 플레이한 신규 유저 비율** (유저 단위) | ≥60% | 40–60% | <40% | D7 |
| 재시도 깊이 (보조) | `game_start` 중 `is_retry=true` 비중 (이벤트 단위 — 헤비 유저 가중) | ≥55% | 35–55% | <35% | D7 |
| 세션당 판수 | 세션 = PostHog 기본 30분 무활동 기준 | ≥4 | 2–4 | <2 | D7 |
| **D1 리텐션** | **D3–D6 유입 오가닉, pooled(Σ복귀/Σ신규), KST 기준** | ≥20% | 10–20% | <10% | **D8** |
| D7 리텐션 | 동일 방식 | ≥6% | 2.5–6% | <2.5% | W2 |
| 고스트 떠 있는 판 % | ghost_count>0 / game_start | ≥95% | 85–95% | <85% | D3·**상시** |
| JS 에러 세션율 | Sentry crash-free rate (Sentry 단독 분모) | <1% | 1–3% | >3% | 상시 |
| 비정상 종료율 | `abnormal_exit` / 세션 (PostHog 단독 분모) — OOM 프록시 | <3% | 3–7% | >7% | 상시 |
| 첫 화면 로드 | p75 (플랫폼 요건 10s) | <4s | 4–8s | >8s | D3 |

- 주지표를 유저 단위로 바꾼 이유: "고스트 추월 → 한 판 더"는 유저 단위 질문인데 이벤트 비중은
  8연속 플레이어 1명이 90%의 1판 이탈을 가릴 수 있다. **주지표가 실패면 코어 루프 반증.**
- 각주1 생존편향: PostHog SDK 로드 전 이탈자는 분모에서 빠진다 → 주간 루틴에서
  "PostHog 부트 진입 ÷ 앱인토스 콘솔 진입" 비율을 계측 건강 지표로 교차 확인.
- **신원 휘발 보정**: localStorage UUID는 웹뷰에서 증발할 수 있다(같은 사람이 신규+이탈로 이중
  왜곡, 하필 회수 판정 쪽으로 편향). "첫 세션인데 숙련 행동(즉시 장거리)" 유저 비율을 휘발
  프록시로 감시, 주 5% 초과 시 리텐션 기준을 그만큼 보정하고 판정 각주에 명기.
  근본책(토스 로그인 연동)은 W4 "계속 투자" 시 최우선 항목.

## 2. 타임라인 — 구간별 상황과 판단

### D-0 (승인일, 배포 전) — 하나라도 안 되면 배포 연기

- [ ] §0 전 항목 완료 확인 (특히 계측 E2E, 코호트 리허설)
- [ ] Supabase 마이그레이션 001·002 프로덕션 적용 + 오늘·내일 시드 봇 확인
- [ ] Sentry 급증 알림 + PostHog **볼륨 급감 알림** + 봇 시드 자동 감시(§7) 켜짐
- [ ] 릴리즈 태그, §1 수치 최종 확정 커밋

**결정 D-0-a. 지인 공유** — D0엔 하지 않는다. D1–D2 소규모(5–10명), §5 절차 필수.

### D0–D3: 기술 안정화 — 게임성 판단 금지

보는 것: Sentry 신규 에러(웹뷰 특이), 부트 퍼널 단계별 이탈, `abnormal_exit`(저사양 OOM —
프레임 이슈면 `RENDER_DPR` 상한 2→1.5), 고스트 떠 있는 판 %, **`game_start`−`game_over` 잔차**
(크면 이벤트 유실 = 이후 지표 전체 불신).

**결정 D0-3-a. 결함 대응 이원화** — "즉시 핫픽스"는 물리적으로 없다 (재검수 영업일 3~5일, 주말 미진행):
- 클라 재배포가 필요한 결함 → **즉시 제출** + 승인까지 **서버측 완화**(§0 원격 스위치로 기능 차단/완화)
- 그 외 전부 → W2 정기 패치로 모은다 (매일 배포하면 지표 원인 귀속 불가)

### D3–D7: 첫 진단

1. §1 추세 확인 (판정 아님)
2. **사망 거리 히스토그램**: 0–100m 밀집=온보딩 / 특정 거리 스파이크=해당 패턴·트랩 벽
   (정전 구간 1,000m+ 스파이크 = 연막 재조정 — 원격 계수면 검수 없이 가능) / 재시도 코호트 우측 이동=건강
3. `fever_count` 분포 — 피버 미경험 유저 비중

**결정 D3-7-a. W2 패치 내용 확정 + 제출 타이밍** — 시즌 동기화는 검수 리드타임을 계산에 넣는다:
**목표 리셋일(월) 기준 최소 7영업일 전 제출.** 승인이 리셋을 넘기면 **다음 주 리셋으로 민다**
(주중에 끼워넣기 금지 — v1/v2 클라가 같은 주간 보드에 섞이는 것 방지. 서버 보드의 SIM_VERSION
파티션 확인이 패치 선행 조건). 배포(승인)와 활성화(리셋일)를 분리할 수 있으면 분리.

### W2: 첫 공식 판정 + 첫 패치

- §1 대조 → §3 시나리오 확정 → 해당 액션만 실행. 패치 전/후 코호트 분리 측정.
- **패치 필수 포함: 시즌 공지 슬롯** (게임오버 화면 카운트다운 — 연출 변경이라 버전 무관).
  공지 채널이 게임 화면뿐이므로, 첫 시즌 리셋(기록 초기화)은 **최소 3일 전부터 인게임 노출**
  없이 단행 금지 — 공지 없는 리셋은 "시즌"이 아니라 데이터 소실로 체감된다.
- **결정 W2-a. 백로그 투자 방향** — 지표가 가리키는 것 **하나만**:
  - 재시도율 실패 → 코어 루프(밸런스·에스컬레이션)만. **경제 착수 금지** (새는 퍼널 위 투자)
  - 재시도율 성공 + D1 실패 → 복귀 훅 (내일 코스 예고, 주간 랭킹 노출, 닉네임)
  - 둘 다 보통 이상 → 아이템/경제 착수 고려

### W4: 4-way 판정

| 판정 | 조건 | 액션 |
|---|---|---|
| **계속 투자** | 핵심 2지표(재시도율·D1) 중 1개 이상 "성공" | 경제·티어 본격화, 노출 확대 협의, 토스 로그인 연동 |
| **개선 사이클** | 핵심 2지표 모두 "보통" | 최악 퍼널 1단계 집중 2주 ×2 → W8 재판정. **사이클은 1회 한정 — W8에 "성공" 0개면 자동 회수, 재재판정 없음** |
| **회수 모드** | 오가닉 D7<2.5% **및** 재시도율<40% **및** 패치 2회 무개선 | §6 → 다음 게임 |
| **표본 부족** | W4까지 오가닉 누적 n<100 | **게임성 판정 없이 S0 확정.** 이후 2주는 노출 작업만(게임 수정 금지) → W6에도 n<100이면 "분포 실패"로 회수 (게임성 미검증 상태임을 §6 보고서에 명기) |

> kill criteria 전에 감정(지침)으로 접는 것도, 충족 후 매몰비용으로 붙잡는 것도 금지.
> W8 종결 규칙이 그 대칭이다 — 애매한 성과가 가장 위험한 지점.

## 3. 시나리오 트리

**S0. 노출 자체가 없다** (일 신규 <50, 기술 정상)
- 검증: 앱인토스 노출 지면 직접 확인, 플랫폼 문의 / 액션: 아이콘·타이틀·설명, 공유 리워드 검토, 지면 협의. **게임 수정 금지**
- 학습: 슈퍼앱 디스커버리 — 다음 입점의 자산

**S0.5. 초기 노출 절벽** (D4~D7 일 신규 급감) — *소프트런치 최다 오판 지점*
- 증상: 신규는 반토막인데 **유입 코호트별** 리텐션은 유지 → 신규 지면 노출 종료일 뿐, 게임 문제 아님
- 검증: 날짜별이 아니라 코호트별로 갈라 보기 / 액션: 기록만, 게임 수정 금지
- §5 보강: 일 신규가 전일 3배 이상 변동한 날 코호트는 별도 태깅, 판정은 태깅 제외/포함 둘 다 기록

**S1. 진입은 있는데 첫 판 시작 안 함** (<60%)
- 검증: 부트 퍼널 단계별 + 콘솔 진입수 교차(생존편향 확인) / 액션: 로드 최적화, 시작 오버레이 단순화

**S2. 첫 판은 하는데 재시도 없음** (주지표 <40%) — **핵심 가설 반증 시나리오**
- 검증 3분기: 첫 판 사망 중간값 <100m → 조작·난이도 / 고스트% 높은데 재시도 없음 → **추월 동기 자체가 안 통함** (최고 가치 데이터) / 게임오버 체류 극단적 짧음 → 결과 화면 문제
- 전제 확인: 기술 게이트 통과 상태에서만 이 판정 유효 (§1 게이트)
- 학습: 이 게임의 존재 가설 실측 답 — 다음 코어 루프 선정의 1차 근거

**S3. 재시도는 하는데 D1에 안 옴** (재시도 정상, D1 <10%)
- 검증: 세션 몰아치기 여부, 이틀째 유저 행동, **KST/UTC 경계**(데일리 시드가 KST 09시 리셋 —
  "내일 새 코스" 훅과 유저의 하루가 어긋남. 시드 리셋 KST 자정 이동은 sim 영향 검토 후 W2 후보)
- 액션: 내일 코스 예고, 주간 랭킹 노출 강화, 닉네임 소유감

**S4. D1은 오는데 D7에 죽음** (D7 <2.5%)
- 검증: 사망 거리 정체(진행감 소멸), 콘텐츠 소진 / 액션: sim 2차 에스컬레이션, 후반 변주, 시즌 보상
- 학습: 순수 메커니즘의 수명 실측

**S5. 지표 좋은데 절대 수 작음** → 게임의 성공, 분포의 실패. §1 표가 플랫폼 협상 카드. 고스트 도전장 공유 투자 가치가 생기는 유일한 시나리오

**S6. 급성장** (일 신규 >5,000)
- 순서: Supabase 티어 한도 → top-N 리플레이 검증 승격 → 봇 커버리지. 결제 한도 사전 결정. "아침에 확인" 원칙

**S7. 리더보드 오염** — *트래픽 무관 상시 리스크 (S6 하위 아님)*
- 유저 200명이어도 콘솔 조작 1명이 주간 1등이면 정직한 상위권부터 이탈 — 가장 가치 있는 코호트가 먼저 죽는다
- 규칙: §0에서 커밋한 임계 초과 기록은 **발견 즉시 보드에서 숨김**(soft-delete, 본인 로컬 표시는 유지) — 논쟁은 사후에

## 5. 코호트 분리 — 지인·사내 트래픽 격리

1인 사내 프로젝트의 최대 오염원은 지인보다 **회사**다 (슬랙 한 번 = 수십 명).

1. 개발자 본인·QA 기기는 PostHog opt-out 플래그로 원천 제외
2. 지인 공유는 **심야 1시간 창**(23:00–24:00)에 몰고, "받는 즉시 열어달라"를 메시지에 명시
3. 창 내 신규 수 ≈ 공유 인원수 대조 — 크게 어긋나면 그 날짜 코호트 전체를 판정 제외
4. 창 내 신규 user_id → PostHog **static cohort** "friends" 저장 (동적 조건 불가 — §0 리허설 필수)
5. 판정 대시보드는 "friends 제외" 필터 저장본만 사용
6. **사내 공지는 W2 판정 이후로 금지**, 하려면 friends와 같은 창 규칙
7. 지인 피드백은 지표가 아니라 정성 데이터 (버그·감상 → 이슈)

표본 규칙: 오가닉 n<100 % 판정 유보, n<30 추세 금지 (히스토그램만 유효). 유보의 출구는 §2 W4 "표본 부족" 행.

## 7. 운영 리듬 — 수동 확인이 아니라 알림 기반

1인 운영자는 "어제와 같아 보이면 넘어가는" 확인 피로에 반드시 걸린다. 자동화가 기본, 수동은 보조.

**자동 알림 (D-0 전 구축)**: Sentry 에러 급증 / PostHog 이벤트 볼륨 급감(사일런트 계측 장애) /
오늘 시드 봇 로그 수 < N (Supabase cron 또는 GitHub Actions 아침 쿼리 — 빈 시드 = 신규 유저 경쟁 필드 증발)

**매일 (10분)**: 알림 없음 확인 / 어제 최고 기록(S7 임계 대조) / 일 신규 (전일 3배 변동 시 코호트 태깅)
**매주 월 (30분)**: 랭킹 리셋 확인, 주간 스냅샷 → `docs/launch-log.md`, PostHog÷콘솔 진입 교차 확인
**W2·W4 (반나절)**: 공식 판정 + 패치

대시보드 위젯: 부트 퍼널 / start·over 추이 + **잔차** / 재시도율(유저·이벤트 둘 다) /
리텐션 매트릭스(friends 제외) / distance 히스토그램 / fever 분포 / ghost_count>0 / abnormal_exit / 신원 휘발 프록시
