# 출시 후 플레이북 (Launch Playbook) — Ghost Dash

작성일: 2026-07-19 · 상태: **GRAC 심사 대기 중** (출시 전 확정할 것)
선행 문서: [toss-miniapp-launch-plan.md](./toss-miniapp-launch-plan.md) (출시 *까지*의 절차) — 이 문서는 출시 *이후*를 다룬다.

> **사전 등록 원칙.** 이 문서의 기준 수치는 출시 전에 확정하고, 판정일 전에 바꾸지 않는다.
> 사후에 보면 어떤 숫자든 합리화된다 — 기준을 미리 박아둬야 실패에서도 배움이 남는다.
> 수치를 바꿔야 하면 커밋으로 남긴다 (변경 이유 포함).

## 1. 판정 기준표 (사전 등록)

기준선 근거: 하이퍼캐주얼 D1 평균 ~26–28%, 상위권 38–40%, D7는 ~8% 수준으로 급락
([AppAgent](https://appagent.com/blog/mobile-game-retention-benchmarks/),
[GameAnalytics 2025](https://www.gameanalytics.com/reports/2025-mobile-gaming-benchmarks)).
단, 이건 **설치형 모바일** 기준. 미니앱은 설치 마찰이 없어 유입 질이 낮고(스쳐가는 유저 다수),
D1이 설치형보다 낮게 나오는 게 정상이라 하향 보정했다.

**모든 % 지표는 오가닉 코호트(§5) 기준. 표본 n<100이면 판정 유보.**

| 지표 (PostHog 소스) | 성공 | 보통 | 실패 | 판정일 |
|---|---|---|---|---|
| 진입→첫 판 시작 전환율 (`game_start` 존재/부트 진입) | ≥80% | 60–80% | <60% | D7 |
| 첫 판 완료율 (첫 `game_over`/첫 `game_start`) | ≥90% | 75–90% | <75% | D7 |
| **자발 재시도율** (`game_start` 중 `is_retry=true` 비중) | ≥55% | 35–55% | <35% | D7 |
| 세션당 판수 | ≥4 | 2–4 | <2 | D7 |
| **D1 리텐션** | ≥20% | 10–20% | <10% | D7 (D1–D6 코호트 평균) |
| D7 리텐션 | ≥8% | 3–8% | <3% | W2 |
| 고스트 떠 있는 판 % (ghost_count>0) | ≥95% | 85–95% | <85% | D3 (기술 지표) |
| 크래시/에러 세션 비율 (Sentry) | <1% | 1–3% | >3% | 상시 |
| 첫 화면 로드 (앱인토스 요건 10s) | <4s p75 | 4–8s | >8s | D3 |

재시도율 기준이 상대적으로 높은 이유: Phase 0 Go/No-Go가 "자발 8연속 플레이"였고,
이 게임의 핵심 가설(고스트 추월 → 한 판 더)이 바로 이 지표이기 때문. **이 지표가 실패면
다른 게 다 좋아도 코어 루프 반증이다.**

## 2. 타임라인 시뮬레이션 — 구간별 상황과 판단

### D-0 (승인일 → 배포 버튼 누르기 전)

체크리스트 — 하나라도 안 되면 배포 연기 (반나절 늦는 건 아무 비용 아님):

- [ ] Supabase 마이그레이션 001(meta)·002(user_id/주간 랭킹 뷰) 프로덕션 적용 확인
- [ ] 오늘·내일 시드 봇 고스트 정상 생성 확인 (원격 `ghost_runs` is_bot 로그)
- [ ] Sentry 알림 채널(에러 급증 알림) + PostHog 대시보드 저장 (아래 §7 위젯 목록)
- [ ] 릴리즈 태그 (예: `v1.0-launch`) — 이후 핫픽스 diff 기준점
- [ ] 이 문서 §1 수치 최종 확정 커밋

**결정 D-0-a. 지인 공유 시점** — 기본값: **D0엔 하지 않는다.** 기술 안정화 전 트래픽은
소음이고, 첫인상 기회를 버그에 소모한다. D1–D2에 5–10명(QA 목적) → §5 절차로 코호트 분리.

### D0–D3: 기술 안정화 구간 — "게임성 판단 금지"

이 구간의 리텐션·재시도 숫자는 지인+호기심 트래픽으로 오염돼 있다. **보는 것은 기술 지표만:**

- Sentry 에러율 / 신규 에러 종류 (특히 앱인토스 WebView 특이 이슈 — 일반 브라우저에서 재현 안 되는 것)
- 부트 퍼널: `setBootLoadingStatus` 단계별 이탈 (화면 준비→폰트→엔진→에셋 중 어디서 새는지)
- 기기 분포 대비 프레임 이슈 신고 — 저사양에서 버벅이면 `RENDER_DPR` 상한 2→1.5 즉시 조정 가능
- 고스트 떠 있는 판 % — 이게 낮으면 Supabase 연결/봇 시딩 문제 (게임성 아님)

**결정 D0-3-a. 핫픽스 기준** — 크래시율 >3% 또는 "게임 진행 불가" 버그: 즉시 핫픽스.
그 외 전부: 모아서 W2 정기 패치. 매일 배포하면 지표의 원인 귀속이 불가능해진다.

### D3–D7: 첫 진단 구간

오가닉 표본이 쌓이기 시작. 할 일:

1. §1 표의 D7 판정 지표들 1차 확인 (아직 판정 아님 — 추세만)
2. **사망 거리 히스토그램** 첫 확인 (`game_over.distance`):
   - 건강한 모양: 첫 판 100–300m 밀집 + 재시도 코호트는 우측으로 이동
   - 0–100m 밀집 = 조작을 이해 못 함 (온보딩 문제) / 특정 거리 스파이크 = 특정 패턴·트랩이 벽 (밸런스 문제)
   - 정전 트랩 구간(1,000m+)에서 스파이크가 서면 연막 난이도 재조정 후보
3. `fever_count` 분포 — 피버를 한 번도 못 본 유저 비중이 크면 FEVER_INTERVAL(10s) 재고

**결정 D3-7-a. 첫 밸런스 패치 내용 확정** — 배포는 W2 랭킹 리셋에 맞춘다 (sim 변경 =
SIM_VERSION 업 = 고스트·보드 무효화이므로, 주간 리셋과 동기화해 "시즌"으로 포장 — TODOS 정책).

### W2: 첫 공식 판정 + 첫 패치

- §1 표와 대조 → §3 시나리오 확정 → 해당 시나리오의 액션만 실행
- 패치 1회 (밸런스 + 모아둔 버그) — 패치 전/후 코호트를 나눠 효과 측정
- **결정 W2-a. 백로그 투자 방향** — 아이템/상점(백로그 5), 닉네임 UI, sim 2차 에스컬레이션 중
  지표가 가리키는 것 **하나만**. 판단 기준:
  - 재시도율 실패 → 코어 루프부터 (에스컬레이션/밸런스). 경제를 얹는 건 금지 — 새는 퍼널에 투자하는 것
  - 재시도율 성공 + D1 실패 → 데일리 복귀 훅 (내일 코스 예고, 주간 랭킹 노출 강화, 닉네임)
  - 둘 다 보통 이상 → 아이템/경제 착수 고려 (LTV 축)

### W4: 계속 / 개선 사이클 / 회수 판정

**결정 W4-a. 3-way 판정:**

| 판정 | 조건 | 액션 |
|---|---|---|
| **계속 투자** | 핵심 2지표(재시도율·D1) 중 1개 이상 "성공" | 백로그 본격 진행 (경제·티어), 노출 확대 협의 |
| **개선 사이클** | 핵심 2지표 모두 "보통" (가장 흔한 결과) | 최악 퍼널 1단계만 집중하는 2주 사이클 ×2 → W8 재판정 |
| **회수 모드** | 오가닉 D7 <3% **및** 재시도율 <35% **및** 패치 2회로 개선 없음 | §6 자산 회수 → 다음 게임. 서비스는 유지보수 모드(데일리 시드·봇은 자동이라 유지 비용 ≈ 0) |

> "회수"는 실패 인정이 아니라 **타율 게임의 다음 타석**이다. 미니앱의 이점은 실험 비용이
> 낮다는 것이고, 그 이점은 죽은 게임을 붙잡지 않을 때만 실현된다. 단, kill criteria를
> 충족하기 전에 감정(지침/조급함)으로 접는 것도 금지 — 그래서 이 표가 미리 있는 것.

## 3. 시나리오 트리 — 증상 → 검증 → 액션 → 학습

**S0. 노출 자체가 없다** (진입 절대수가 극소)
- 증상: 일 신규 진입 <50, 기술 지표는 정상
- 검증: 앱인토스 내 검색/카테고리 노출 위치 직접 확인, 플랫폼 담당에게 노출 지면 문의
- 액션: 아이콘·타이틀·설명 개선, 공유 리워드 SDK 검토, 플랫폼 이벤트/추천 지면 협의. **게임 수정 금지**
- 학습: 슈퍼앱 디스커버리 메커니즘 — 다음 게임의 입점 전략 자산

**S1. 진입은 있는데 첫 판을 시작 안 한다** (전환율 <60%)
- 검증: 부트 퍼널 단계별 이탈 지점 + 로드 시간 p75 + 시작 오버레이에서 이탈하는지
- 액션: 로드 최적화(번들 스플릿), 시작 오버레이 문구/CTA 단순화
- 학습: 미니앱 유저의 인내 한계 — "설치 없는 유입은 3초 안에 잡아야 한다"의 실측치

**S2. 첫 판은 하는데 재시도가 없다** (재시도율 <35%) — **핵심 가설 반증 시나리오**
- 검증 (원인 3분기):
  - 첫 판 사망 거리 중간값 <100m → 조작/난이도 문제 (게임이 어렵다고 느끼기 전에 죽음)
  - 고스트 떠 있는 판 %는 높은데 재시도 없음 → **고스트 추월 동기 자체가 안 통함** (뼈아프지만 최고 가치 데이터)
  - 게임오버 화면 체류 시간 극단적으로 짧음 → 결과 화면이 재도전 동기를 못 만듦 ("ONE MORE RUN?" 노출률 확인)
- 액션: 분기별로 각각 — 온보딩 램프 완화 / 추월 연출·근접 경쟁 강화(아깝게 진 판 연출) / 결과 화면 개선
- 학습: 이 게임의 존재 이유였던 "고스트 경쟁 = 재시도" 가설의 실측 답. 다음 게임 코어 루프 선정의 1차 근거

**S3. 재시도는 하는데 다음 날 안 온다** (재시도율 정상, D1 <10%)
- 검증: 세션 길이 분포 (한 번에 몰아서 다 소진?), 데일리 시드 인지 여부 (이틀째 온 유저의 행동)
- 액션: "내일은 새 코스" 예고를 게임오버 화면에 명시, 주간 누적 랭킹 노출 강화(내 순위 변동), 닉네임 UI로 소유감
- 학습: 세션 내 재미와 복귀 훅은 별개 근육이라는 것 — 하이퍼캐주얼의 고전적 약점(D7 ~8%)을 데일리 시드가 얼마나 보완하는지 실측

**S4. D1은 오는데 D7에 죽는다** (D1 정상, D7 <3%)
- 검증: 이탈 직전 코호트의 사망 거리 정체 여부 (실력 정체 = 진행감 소멸), 콘텐츠 소진(바이옴 4종 다 봄)
- 액션: sim 2차 에스컬레이션(1.15.0), 후반 콘텐츠(장애물·트랩 변주), 주간 시즌 보상
- 학습: 코어 루프의 수명 — 콘텐츠 없이 순수 메커니즘으로 버티는 일수

**S5. 지표는 좋은데 절대 수가 작다**
- 판정: 이건 게임의 성공, 분포의 실패. §1 표를 들고 플랫폼 노출 협의 (지표가 협상 카드)
- 액션: 공유 기능(고스트 도전장 — "내 고스트 이겨봐" 링크) 투자 가치가 생기는 유일한 시나리오

**S6. 예상 밖 급성장** (일 신규 >5,000)
- 리스크 순서: Supabase 무료 티어 한도(요청 수·저장량 — ghost_runs 증가율 확인) → 리더보드
  오염(이상치 필터 상회하는 정교한 조작 — top-N 리플레이 검증 승격) → 봇 시드 커버리지
- 미리 정해둘 것: 티어 업그레이드 결제 한도, 온콜 아닌 "아침에 확인" 원칙 (미니게임은 야간 장애로 죽지 않는다)

## 5. 코호트 분리 — 지인 트래픽이 지표를 오염시키지 않게

문제: 지인은 D1 복귀·재시도가 오가닉보다 훨씬 높거나(의리) 낮다(한 번 보고 끝). n이 작은
초기엔 이게 전체 지표를 지배한다.

방법 (미니앱은 URL 파라미터 전달이 불확실하므로 **시간 창 + user_id 목록** 방식):

1. 지인 공유는 **특정 시간 창에 몰아서** 한다 (예: D1 20:00–24:00)
2. 그 창에서 생성된 신규 `user_id`(localStorage UUID, PostHog person 연동) 목록을 뽑아 PostHog Cohort "friends"로 저장
3. 모든 판정 지표는 **"friends 제외" 필터를 기본값**으로 저장된 대시보드에서만 읽는다
4. 지인 피드백은 지표가 아니라 **정성 데이터**로 취급 (버그·감상 → 이슈로)

표본 규칙: 오가닉 n<100 → % 지표 판정 유보, n<30 → 추세 언급도 금지 (사망 분포 히스토그램만 유효).

## 6. 회수 모드 체크리스트 — 실패해도 남기는 것

- [ ] **실측 데이터 아카이브**: 퍼널 수치 스냅샷 → 다음 게임의 §1 기준치는 이 실측치로 교체
- [ ] **판정 보고서 1p** (사내): 어느 시나리오였나 / 반증·입증된 가설 / 다음 게임 제안
- [ ] **재사용 자산 목록 확정**: 결정론 sim 코어 + 고스트/리플레이 + 봇 생성기 + remoteStore/identity +
      계측 파이프라인 + 앱인토스 입점·GRAC 서류 템플릿 — 다음 게임은 "코어 루프만 교체"로 착수
- [ ] 유지보수 모드 전환: 데일리 시드·봇·랭킹은 무인 동작 확인, Supabase 비용 모니터만 남김
- [ ] 회고: "이 문서의 어떤 기준이 틀렸었나" — 플레이북 자체를 다음 게임용으로 개정
