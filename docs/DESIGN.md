## Problem Statement

블록체인 게임 퍼블리싱 회사가 1인 프론트엔드 개발자 + 생성형 AI(코드, 에셋, 게임디자인 전부)만으로 미니게임을 자체 개발한다. 개발자는 게임 개발 경험이 없다.

**플랫폼 (2026-07-06 확정):** 토스 미니앱을 1차 배포 타깃으로 결정. 자사앱 배포는 열리면 보너스, Vercel(현재 ghost-arcade-five.vercel.app)은 개발/테스트 채널 유지.

## Demand Evidence

- 솔직한 현황: **검증된 유저 수요는 아직 없음.** 사업부와 플랫폼 개발자들의 관심은 내부 이해관계자 신호이지 플레이어 수요가 아님.
- 첫 번째 게임(쓰리매치)은 아직 런칭 전이라 참고할 유저 데이터가 없음.
- 개발자 본인의 프레이밍: "게임의 성패는 알 수 없으니, 우선 유저 데이터를 충분하게 쌓을 수 있는 게임을 만들자" — 학습 목적의 실험으로 포지셔닝됨.

## Status Quo

- 회사의 기존 방식은 외부 게임을 플랫폼에 온보딩시키는 퍼블리싱. 자체 개발은 이번이 처음.
- 토큰/포인트 보상 연동 없음 — 게임은 순수하게 게임 자체의 재미로만 작동해야 함.
- 토스 미니앱(앱인토스)을 통한 외부 배포. 회사 자체 슈퍼앱 배포는 가능성 낮음.

## Target User & Narrowest Wedge

- **타겟:** 토스 앱 유저. 금융 용무로 앱을 열었다가 잠깐의 여유 시간에 미니게임을 접하는 사람. 토스의 만보기·출석 등 "매일 들어오는" 습관 문화와 데일리 시드 루프가 구조적으로 맞음.
- **미해결 과제:** 구체적인 페르소나(연령대, 사용 패턴, 게임 성향)가 아직 정의되지 않음.
- **가장 좁은 웻지:** 탭 한 번으로 조작하는 엔들리스 아케이드 코어 1개 + 점수 기록. 고스트는 코어 검증 후 2단계로 추가.

## Constraints

- 개발 인력: 프론트엔드 개발자 1인, 게임 개발 경험 없음.
- 모든 제작(코드, 에셋, 게임디자인)을 생성형 AI로 수행.
- 배포 형태: 토스 미니앱(앱인토스) WebView.
- 보상 시스템(토큰/포인트) 없음 — 게임 자체의 재미로 리텐션을 만들어야 함.
- 첫 게임(쓰리매치) 미런칭 상태라 참고할 유저 데이터 없음.

## Premises

1. **개발 형태는 HTML5 게임(WebView).** Unity MCP가 아닌 Phaser.js 등 웹 기술 스택. Unity는 빌드 용량·WebView 호환성·배포 파이프라인이 1인 개발에 과함. — 동의됨
2. **"어떤 장르든 가능"은 아니다.** 1인 + AI 조합으로 현실적인 장르(아케이드, 퍼즐, 클리커 등)로 범위를 좁혀야 한다. — 동의됨
3. **보상이 없으면 게임 자체가 즉각적으로 재미있어야 한다.** '한 판 더' 루프가 코어에 있어야 함. — 동의됨
4. **데이터는 사람이 실제로 플레이해야 쌓인다.** "데이터 수집용 게임"이 아니라 "계속 하고 싶은 게임"이 먼저다. — 동의됨

## Approaches Considered

### Approach A: 데일리 퍼즐 (워들 스타일)
매일 1판, 제한된 시도로 정답을 맞추는 퍼즐. Effort: S / Risk: Low. 세션이 짧아 페이앱 맥락에 맞고 DAU 측정이 깔끔하지만, 하루 1판이라 데이터 밀도가 낮고 한국어 콘텐츠 생성이 복잡함.

### Approach B: 엔들리스 아케이드 (단독)
탭 하나로 조작, 버티기 경쟁. Effort: S / Risk: Low. 개발 난이도 최저, 데이터 풍부. 그러나 개발자 본인이 "너무 단순해서 흥미가 안 생긴다"고 밝힘 — 1인 개발에서 제작 동기 상실은 실질적 리스크.

### Approach C: 탭 클리커 / 방치형
체크인 습관 유도. Effort: M / Risk: Medium. 보상 연동 없이는 장기 리텐션이 약하고 시스템 설계가 복잡함.

### Approach D: AI 심판 게임 (데일리 챌린지형)
AI가 출제하고 채점하는 게임(프롬프트 배틀 등). Effort: M / Risk: Medium. "AI로만 만들 수 있는 게임"이라는 차별화가 명확하나 LLM API 비용, 응답 지연, 모더레이션이라는 운영 숙제가 있음. 세션에서 추천되었으나 미채택.

### Approach E: 고스트 배틀 아케이드 (채택)
엔들리스 아케이드에 다른 유저의 고스트(과거 플레이 기록 재생)를 띄워 비동기 경쟁. Effort: M / Risk: Medium. 실시간 서버 없이 멀티 느낌 구현. 고스트 경쟁은 유저 밀도가 있어야 작동 → 토스 MAU가 그 밀도를 공급. 단, 게임 코어가 재미없으면 고스트도 무용지물이고, 초기 유저 풀이 작으면 고스트 매칭이 빈약함.

### Approach F: AI 생성 데일리 내러티브
매일 AI가 만드는 선택지 스토리. Effort: M-L / Risk: High. 코딩 난이도는 낮지만 스토리 품질 일관성 유지가 어려움.

## Recommended Approach

**Approach E: 고스트 배틀 아케이드** (사용자 선택)

탭/클릭 하나로 조작하는 엔들리스 아케이드에 **다른 유저의 고스트(과거 플레이 기록 재생)**를 띄워 비동기 경쟁을 만든다. 실시간 멀티플레이 서버 없이 멀티 느낌을 구현한다.

선택 이유:
- B의 낮은 개발 난이도(완성 가능성)를 유지하면서 소셜 경쟁 레이어로 차별화와 리텐션을 보완.
- "같은 날 같은 코스를 달린 사람들과 경쟁"이라는 구도가 토스 유저 기반 + 데일리 습관 문화와 시너지.
- 고스트 재생은 입력 로그 기록/재생으로 구현 가능 — 프론트엔드 개발자의 기존 역량 범위 안.

단계적 구축 (중요):
1. **1단계 (1-2주):** 아케이드 코어 + 기본 계측. 탭 조작, 점수, 게임오버, 재시작 루프. 세션 수·판수·재시도율 계측을 이 단계에 포함한다(고스트 도입 전 베이스라인 확보 목적). 이 단계에서 코어가 재미있는지 검증한다. **코어가 재미없으면 고스트도 무용지물** — 이 리스크가 이 설계의 최대 약점.
   - **주의:** 1단계의 산출물은 기각한 Approach B와 겉보기에 같다. 차이는 (a) 처음부터 고스트를 전제로 한 결정론적 시뮬레이션 설계, (b) 계측 내장. 그래도 "단순해서 흥미가 안 생긴다"던 동기 리스크가 1단계에 그대로 적용된다는 점은 인정하고 시작할 것 — 2단계가 있다는 사실이 동기를 지탱해야 한다.
   - **Go/No-Go:** 내부 테스트에서 성공 기준 미달 시 고스트(2단계)로 넘어가지 말 것. 코어 컨셉을 교체해 1단계를 반복한다. 2회 교체 후에도 미달이면 장르 재검토(Approach D 재고려).
2. **2단계 (플랫폼 API 스펙 수령 후 일정 확정, 순수 개발 1-2주):** 고스트 시스템. 입력 로그 저장(타임스탬프 + 입력 이벤트), 고스트 재생, 전체 리더보드 고스트 선택. 친구 고스트는 토스의 소셜 그래프/친구 API 제공 여부 확인 후 별도 단계로 분리. **폴백:** 플랫폼 백엔드 스펙이 지연되면 로컬 시드 고스트(개발자/팀 플레이 기록 번들)만으로 2단계 검증을 진행한다.
   - **Go/No-Go:** 고스트 도입 후 세션당 플레이 횟수가 베이스라인 대비 증가하지 않으면, 고스트 노출 방식을 1회 조정(예: 오버레이 → 막판 추월 연출 강조)하고 재측정한다. 그래도 미달이면 "고스트가 리텐션을 보완한다"는 핵심 가설이 기각된 것 — 고스트 없이 단독 코어로 런칭할지, 다른 소셜 장치(점수 공유 등)로 교체할지 사업부와 판단한다.
3. **3단계:** 계측 고도화. D1/D7 리텐션 코호트, 고스트 도입 전후 세션당 플레이 횟수 비교(1단계 베이스라인 대비).

기술 스택 (제안):
- 렌더링: Phaser.js 3 (WebView 호환성 검증된 HTML5 게임 엔진)
- **결정론 보장 (필수):** 고스트 재생은 같은 입력 → 같은 결과가 보장될 때만 성립한다. 고정 타임스텝(fixed timestep) + 시드 고정 RNG(seeded RNG)로 시뮬레이션을 결정론적으로 설계한다. 이것이 불가능한 구조라면 N프레임 간격 위치 스냅샷 로깅 방식으로 대체 — 단, **스냅샷 폴백을 택하면 서버 측 리플레이 재검증(Open Question 6)이 불가능해지고 변조 대응은 이상치 필터링만 남는다**는 트레이드오프를 감수해야 함. 1단계 코어 설계 시점부터 반영해야 함 — 나중에 리팩터링하면 비용이 큼.
- **고스트 로그 버전 호환성:** 결정론적 재생은 "같은 시뮬레이션 코드"가 전제다. 게임 밸런스/물리를 패치하면 기존 고스트 로그가 무효화된다(같은 입력을 재생해도 다른 결과). 고스트 로그에 시뮬레이션 버전 태그를 넣고, 버전 불일치 시 해당 고스트는 재생 목록에서 제외한다. 1단계에서 코어 컨셉 교체를 반복할 계획이므로 이 정책은 처음부터 필요함.
- 고스트 데이터: 입력 이벤트 로그 (JSON, 판당 수 KB 수준) + 시뮬레이션 버전 태그
- 백엔드: 점수/고스트 저장용 경량 API (Supabase dumb store — 현재 라이브. 토스 신원 훅 연동 방식 별도 협의)
- 에셋: 생성형 AI. 단, 여러 에셋 간 스타일 일관성 유지가 실제로는 까다로우므로(특히 애니메이션 프레임) **단일 캐릭터 + 도형/파티클 위주의 미니멀 비주얼**로 에셋 수 자체를 줄이는 방향을 권장.
- 비기능 정책: 오프라인 시 로컬 플레이 가능 + 점수 미제출(재접속 시 동기화), 앱 백그라운드 전환 시 자동 일시정지.

## 플랫폼 전환 & 인앱 경제 (2026-07-06 확정)

### ★ 플랫폼: 자사앱 → 토스 미니앱 1차 배포

**이유 (한 줄):** 이 게임의 핵심 가치(크로스유저 고스트 + 데일리 리더보드)는 유저 밀도가 있어야 작동하고, 토스가 그 밀도를 공급하는 유일한 채널.

추가 시너지:
- 데일리 시드 루프 ↔ 토스 "만보기·출석" 매일 들어오는 습관 문화
- 토스 실명 금융앱 = 시빌 방어 (자사앱의 페이앱 가입보다 오히려 더 강함)
- 세션 모양(30초~2분, 탭 조작) = 토스 미니앱 소비 패턴 정확히 맞음

**자사앱 전제로 설계됐던 것들이 바뀌는 것 (인지하고 진행):**

1. **블록체인 토큰 보상 thesis → 소멸.** 기존 thesis("결정론 코어 = 치팅방지 = 토큰 보상 무결성"이 왜 블록체인 슈퍼앱인가의 답)는 토스에선 성립 안 함. 코어 기술(결정론, 서버 리플레이)은 유효. 이 thesis는 자사앱이 열릴 때 재가동.
2. **신원 훅 교체 필요.** 슈퍼앱 계정 연동 → 토스 계정/로그인 API. 백엔드가 KVStore 인터페이스로 추상화돼 있어 갈아끼우기 용이.
3. **인앱 포인트 경제 = 토스 정책 종속.** 토스가 허용하는 범위까지만 가능. 기술 설계는 단순 닫힌 경제(포인트)로 유지.
4. **사업부 배포 승인 필요.** 자사앱 아닌 외부 채널(토스) 배포에 대한 회사 내부 결재.
5. **iOS JSC 결정론 검증이 더 급해짐.** 토스 유저 iOS 비중이 높아 크로스유저 고스트 신뢰 전 실기기 검증 필수.

### 인앱 경제 방향 (단순화)

- **기존:** 블록체인 토큰 faucet/sink + 티어 보상 시스템 (자사앱 전제)
- **변경:** 닫힌 인앱 포인트만. 포인트로 캐릭터/부스터 구매. 토큰/체인 연결 없음.
- **유지:** 닫힌 경제(외부 현금화 불가) + 규제/도박 리스크 해소는 그대로.
- **향후:** 자사앱 배포 시 블록체인 토큰 thesis 재가동. 토스 내에서 토스포인트 연동은 토스 정책 심사 후.

**MVP 스코프 (불변):** 단일 데일리 보드 + 데이터 수집. 백엔드엔 신원 훅 자리만 예약.

## Open Questions

1. **토스 유저의 구체적 프로필** — 연령대, 사용 패턴, 게임 성향. 앱인토스 입점 후 PostHog 데이터로 확인.
2. **고스트 매칭 풀** — 초기 유저가 적으면 고스트가 빈약함. 시드 고스트(개발자/팀 플레이 기록 또는 AI 생성 고스트)로 보완할지?
3. **토스 연동 방식** — 토스 미니앱 SDK 스펙, 유저 식별(토스 계정 로그인 API) 방식. 진입 체크리스트 "기술/플랫폼" 항목 참고.
4. **WebView 성능 제약** — 대상 기기 사양과 WebView 버전 범위. 저사양 기기에서 60fps 유지 가능한지.
5. **아케이드 코어의 구체적 컨셉** — "탭 게임"의 테마/비주얼/조작감은 미정. 1단계 시작 전 2-3개 코어 프로토타입을 만들어 비교하는 것을 권장. 비교 기준: (a) 테스터의 자발적 재시도율, (b) 첫 판 사망까지의 시간이 너무 짧지 않은지(조작 학습 가능성).
6. **점수/고스트 변조 대응** — WebView 클라이언트가 점수와 고스트 로그를 직접 제출하는 구조라 조작 가능. 실명 금융앱(토스) 내 리더보드라 신뢰성 훼손 시 평판 리스크가 일반 미니게임보다 큼. 서버 측 고스트 로그 재검증(리플레이 시뮬레이션) 또는 이상치 필터링 중 어느 수준까지 할지 결정 필요. 단, 서버 리플레이 검증은 결정론 설계만으로 자동으로 따라오지 않음 — 시뮬레이션 로직을 렌더링(Phaser)과 분리해 headless 환경에서 실행 가능하게 만드는 추가 설계가 1단계 아키텍처부터 필요함.
7. **고스트의 게임플레이 의미론** — 고스트가 시각적 오버레이(유령처럼 겹쳐서 달리기만 함)인지, 충돌/상호작용이 있는지 미정. 오버레이 방식이 결정론 요구 범위가 좁고 구현이 단순하므로 기본값으로 권장하되, 코어 컨셉 확정 시 함께 결정할 것.
